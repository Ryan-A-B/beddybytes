# MQTT Topics

This document describes the BeddyBytes MQTT topic contract used for client
presence, WebRTC signalling, and station discovery.

Frontend migration goal: after the direct-MQTT frontend update, the app should
use the backend only for account creation and login. Session discovery and the
parent station session list should be built from MQTT station discovery messages
instead of backend session-list HTTP requests.

Source-backed references:

- Topic helpers: `golang/internal/backendmqtt/Topics.go`
- Payload structs: `golang/internal/backendmqtt/Payloads.go`
- Current backend sync behavior: `golang/internal/backendmqtt/Sync.go`
- Current backend/WebSocket bridge: `golang/cmd/backend/Connection.go`
- Contract tests: `integration_tests/src/tests/test_backend_mqtt.py`

## Conventions

- All topic path IDs are scoped to one account.
- In per-client topics, `{client_id}` must match the MQTT client ID for the
  connected client. This is enforced by the MQTT authorizer for self-scoped
  subscriptions and client status publishing.
- `account_id`, `client_id`, `connection_id`, `session_id`, and `request_id`
  are opaque strings.
- MQTT routing is by `account_id` and `client_id`. `connection_id` is not used
  for MQTT routing in the direct-MQTT frontend; it remains in payloads for
  deduplication, analytics, and compatibility.
- `request_id` is generated by the frontend client on every MQTT connect or
  reconnect. It is used for deduplication, analytics, and stale lifecycle
  cleanup. On reconnect, the client publishes a new `connected` status payload
  with a new `request_id`.
- If a baby station reconnects while its local session is still running, it
  publishes a fresh baby station announcement after reconnect. Parent stations
  do not maintain special state for unexpectedly disconnected baby stations;
  they blindly apply status removals and station announcements.
- Sessions are immutable. If a parent station receives a duplicate
  `session_id`, it treats it as the same session and drops the duplicate.
- `at_millis` and `started_at_millis` are Unix timestamps in milliseconds.
- Current backend publishers use QoS 1 and non-retained messages.
- Payloads are JSON objects.
- Consumers should ignore unknown JSON fields for forward compatibility.

## Topic Summary

| Topic | Purpose |
| --- | --- |
| `accounts/{account_id}/clients/{client_id}/status` | A client announces connection lifecycle state. |
| `accounts/{account_id}/clients/{client_id}/webrtc_inbox` | A client receives WebRTC offer, answer, and ICE candidate messages. |
| `accounts/{account_id}/clients/{client_id}/control_inbox` | A client receives control-plane messages, currently baby station announcements from baby stations. |
| `accounts/{account_id}/baby_stations` | Baby stations announce active sessions. |
| `accounts/{account_id}/parent_stations` | Parent stations announce themselves so active baby stations can be sent to them. |

## `accounts/{account_id}/clients/{client_id}/status`

Client lifecycle topic. The backend currently subscribes to
`accounts/+/clients/+/status` and uses the account and client IDs from the topic
path as the authoritative scope.

All clients in an account should subscribe to
`accounts/{account_id}/clients/+/status` so they can observe which clients are
online. Clients may only publish their own status topic:
`accounts/{account_id}/clients/{mqtt_client_id}/status`.

### Connected Payload

Published when a client connection is available.

```json
{
  "type": "connected",
  "connection_id": "connection-123",
  "request_id": "request-123",
  "at_millis": 1761800000000
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `type` | Yes | Literal value `connected`. |
| `connection_id` | Yes | Opaque ID for deduplication, analytics, and compatibility. Not used for direct-MQTT routing. |
| `request_id` | Yes | Frontend-generated ID for this MQTT connect/reconnect attempt. Used for deduplication, analytics, and stale disconnect cleanup. |
| `at_millis` | Yes | Publish time as Unix milliseconds. |

Current backend behavior:

- Records the client as connected under both `client_id` and `connection_id`.
- Emits a `client.connected` event.
- If an HTTP-created baby station session is pending for this `connection_id`,
  publishes an announcement to `accounts/{account_id}/baby_stations`.

### Disconnected Payload

Published when a client connection is no longer available.

```json
{
  "type": "disconnected",
  "connection_id": "connection-123",
  "request_id": "request-123",
  "at_millis": 1761800000000,
  "disconnected": {
    "reason": "clean"
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `type` | Yes | Literal value `disconnected`. |
| `connection_id` | Yes | Opaque ID for deduplication, analytics, and compatibility. |
| `request_id` | Yes | Must match the active connection request to remove it from the registry. |
| `at_millis` | Yes | Publish time as Unix milliseconds. |
| `disconnected.reason` | Yes | Known values are `clean` and `unexpected`. |

Reason semantics:

- `unexpected` is sent by the MQTT broker from the client's Last Will and
  Testament when the client disconnects unexpectedly.
- `clean` is sent by the client when it shuts down gracefully.
- Explicit user Stop should send `clean` before MQTT disconnect. Browser or tab
  shutdown does not need special clean-disconnect handling; if the MQTT client
  sends a protocol-level DISCONNECT, the broker will not send the Last Will and
  Testament, and if the WebSocket disappears without protocol-level DISCONNECT,
  the broker should publish the Last Will and Testament after detecting the
  connection loss.

Current backend behavior:

- Removes the connection only if `connection_id` and `request_id` match the
  active registry entry.
- Emits a `client.disconnected` event.
- Session projections remove sessions hosted by the disconnected connection.

Frontend direct-MQTT behavior:

- The parent station session list represents baby stations that are currently
  MQTT-connectable. Any baby station disconnect, clean or unexpected, removes
  that baby station from the session list.
- Active viewing is separate from session-list membership. If the parent station
  is currently watching a baby station and sees that baby station publish a
  clean disconnect, it should stop watching.
- If the parent station is currently watching a baby station and sees that baby
  station disconnect unexpectedly, it should not stop watching solely because of
  MQTT status. The WebRTC connection is independent of both clients' MQTT
  connections.

## `accounts/{account_id}/clients/{client_id}/webrtc_inbox`

Per-client WebRTC signal inbox. The frontend should publish WebRTC signals to
the target client's inbox. The receiving client subscribes to its own inbox.
Messages are routed by the target `client_id` in the topic, not by
`connection_id`.

WebRTC role rule: the parent station always sends the WebRTC offer to the baby
station. The baby station answers the parent station's offer and never initiates
an offer to the parent station.

### Description Payload

Used for WebRTC offers and answers.

```json
{
  "from_client_id": "sender-client-123",
  "type": "description",
  "description": {
    "type": "offer",
    "sdp": "..."
  }
}
```

The same envelope is used for answers:

```json
{
  "from_client_id": "sender-client-123",
  "type": "description",
  "description": {
    "type": "answer",
    "sdp": "..."
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `from_client_id` | Yes | Client ID of the sender. The current backend uses this to look up the sender connection. |
| `type` | Yes | Literal value `description`. |
| `description` | For offers/answers | Raw `RTCSessionDescriptionInit`-compatible JSON. |
| `candidate` | No | Omitted for description messages. |

### Candidate Payload

Used for WebRTC ICE candidates.

```json
{
  "from_client_id": "sender-client-123",
  "type": "candidate",
  "candidate": {
    "candidate": "candidate:...",
    "sdpMid": "0",
    "sdpMLineIndex": 0
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `from_client_id` | Yes | Client ID of the sender. |
| `type` | Yes | Literal value `candidate`. |
| `candidate` | For ICE candidates | Raw `RTCIceCandidateInit`-compatible JSON. |
| `description` | No | Omitted for candidate messages. |

Current backend/WebSocket bridge behavior:

- WebSocket signal messages are converted to this MQTT payload shape.
- Incoming MQTT messages are converted back to WebSocket signal messages.
- The current backend does not read the `type` discriminator, so adding it does
  not require a backend change as long as `description` or `candidate` remains
  present.
- When bridging MQTT to WebSocket, the backend currently resolves
  `from_client_id` to `from_connection_id`. Direct frontend MQTT consumers will
  need to use `from_client_id` unless a future payload adds `from_connection_id`.

Direct-MQTT frontend behavior:

- Send WebRTC signals by publishing to the target client's `webrtc_inbox`.
- Route by `client_id`; do not use `connection_id` for routing.
- Parent stations send `description.type: "offer"` messages to baby stations.
- Baby stations send `description.type: "answer"` messages to parent stations.
- Either side may send ICE candidate messages after the offer/answer exchange
  begins.
- Treat `from_client_id` as the sender identity within the account. Spoofing is
  not a concern for this application because clients are already constrained to
  their own account namespace.

Review note:

- The current code classifies payloads with WebRTC `type` values `offer` or
  `answer` as `description`; anything with a `candidate` field is classified as
  `candidate`; otherwise it falls back to `candidate`.

## `accounts/{account_id}/clients/{client_id}/control_inbox`

Per-client control-plane inbox. The backend currently publishes known baby
station sessions to a parent station after the parent announces itself for
legacy frontend clients. The target direct-MQTT behavior is for baby stations to
subscribe to `accounts/{account_id}/parent_stations` and publish their own
announcement to the new parent station's `control_inbox`.

### Baby Station Announcement Payload

```json
{
  "type": "baby_station_announcement",
  "at_millis": 1761800000000,
  "baby_station_announcement": {
    "client_id": "baby-client-123",
    "connection_id": "baby-connection-123",
    "session_id": "session-123",
    "name": "Nursery",
    "started_at_millis": 1761800000000
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `type` | Yes | Literal value `baby_station_announcement`. |
| `at_millis` | Yes | Publish time as Unix milliseconds. |
| `baby_station_announcement.client_id` | Yes | Client ID for the baby station. |
| `baby_station_announcement.connection_id` | Yes | Current connection ID for the baby station. |
| `baby_station_announcement.session_id` | Yes | Active baby station session ID. |
| `baby_station_announcement.name` | Yes | User-facing baby station/session name. |
| `baby_station_announcement.started_at_millis` | Yes | Session start time as Unix milliseconds. |

## `accounts/{account_id}/baby_stations`

Account-wide baby station announcement topic. Baby stations publish here when
they have an active session that parent stations may join.
Online parent stations subscribe to this topic, add announced baby stations to
their session list, and if they are not currently viewing a baby station session,
immediately attempt to establish a WebRTC connection.

### Announcement Payload

```json
{
  "type": "announcement",
  "at_millis": 1761800000000,
  "announcement": {
    "client_id": "baby-client-123",
    "connection_id": "baby-connection-123",
    "session_id": "session-123",
    "name": "Nursery",
    "started_at_millis": 1761800000000
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `type` | Yes | Literal value `announcement`. |
| `at_millis` | Yes | Publish time as Unix milliseconds. |
| `announcement.client_id` | Yes | Client ID for the baby station. |
| `announcement.connection_id` | Yes | Current connection ID for the baby station. |
| `announcement.session_id` | Yes | Active baby station session ID. |
| `announcement.name` | Yes | User-facing baby station/session name. |
| `announcement.started_at_millis` | Yes | Session start time as Unix milliseconds. |

Current backend behavior:

- Subscribes to `accounts/+/baby_stations`.
- Accepts only `type: "announcement"`.
- Appends a `session.started` event and updates the baby station list projection.

Frontend target behavior:

- Parent stations subscribe to `accounts/{account_id}/baby_stations`.
- When a baby station announcement is observed, parent stations add it to their
  session list.
- If a baby station reconnects after an unexpected MQTT disconnect, it publishes
  another announcement and parent stations add it back to their session list.
- If a parent station is not currently viewing a baby station session, it should
  immediately attempt to establish a WebRTC connection to the newly announced
  baby station by sending the offer.
- Parent stations should not request the backend session list once the
  direct-MQTT frontend path is complete.

## `accounts/{account_id}/parent_stations`

Account-wide parent station announcement topic. Parent stations publish here to
announce that they are online and want the current active baby station list.
Active baby stations should subscribe to this topic and respond by publishing a
`baby_station_announcement` message to the announcing parent's `control_inbox`.
This gives newly opened parent stations the current set of already-online baby
stations without requiring a backend session-list request.

### Announcement Payload

```json
{
  "type": "announcement",
  "at_millis": 1761800000000,
  "announcement": {
    "client_id": "parent-client-123",
    "connection_id": "parent-connection-123"
  }
}
```

Fields:

| Field | Required | Notes |
| --- | --- | --- |
| `type` | Yes | Literal value `announcement`. |
| `at_millis` | Yes | Publish time as Unix milliseconds. |
| `announcement.client_id` | Yes | Client ID for the parent station. Used to address its `control_inbox`. |
| `announcement.connection_id` | Yes | Current connection ID for deduplication, analytics, and compatibility. Not used for direct-MQTT routing. |

Current backend compatibility behavior:

- Subscribes to `accounts/+/parent_stations`.
- Accepts only `type: "announcement"`.
- Reads the current baby station list projection.
- Publishes one `baby_station_announcement` control message to the announcing
  parent's `control_inbox` for each active baby station.

Target direct-MQTT behavior:

- Baby stations subscribe to `accounts/{account_id}/parent_stations`.
- When a parent station announcement is observed, each active baby station
  publishes its own `baby_station_announcement` payload to
  `accounts/{account_id}/clients/{parent_client_id}/control_inbox`.
- Parent stations merge baby station announcements from MQTT into the frontend
  session list.
- After updated clients are fully rolled out, backend signalling and station
  announcement fan-out can be deprecated and removed from the critical path.

## Authorization Implications

The custom MQTT authorizer should grant access only within the authenticated
account's topic namespace.

Likely client permissions:

| Operation | Topic or topic filter |
| --- | --- |
| Connect | Client connects with an application-selected MQTT client ID. |
| Publish | `accounts/{account_id}/clients/{mqtt_client_id}/status` only. |
| Publish | `accounts/{account_id}/clients/+/webrtc_inbox` account-scoped so clients can signal each other. |
| Publish | `accounts/{account_id}/clients/+/control_inbox` account-scoped so baby stations can announce themselves to parent stations. |
| Publish | `accounts/{account_id}/baby_stations` |
| Publish | `accounts/{account_id}/parent_stations` |
| Subscribe/Receive | `accounts/{account_id}/clients/{mqtt_client_id}/webrtc_inbox` only. |
| Subscribe/Receive | `accounts/{account_id}/clients/{mqtt_client_id}/control_inbox` only. |
| Subscribe/Receive | `accounts/{account_id}/clients/+/status` to observe account client presence. |
| Subscribe/Receive | `accounts/{account_id}/baby_stations` if parent stations consume account-wide baby station announcements directly. |
| Subscribe/Receive | `accounts/{account_id}/parent_stations` for baby stations observing new parent stations. |

Review note:

- The authorizer must enforce `{client_id} == ${iot:ClientId}` for self-scoped
  per-client subscriptions and client status publishing.
- Publish permission for `webrtc_inbox` and `control_inbox` must remain
  account-scoped, not self-scoped, because clients publish messages to other
  clients' inboxes.
- Account-wide client status visibility is intentional. Clients are constrained
  to their own account namespace, and the expected number of concurrently online
  clients is small.
